What Exactly Is A Closure?

A closure is a function that remembers the variables from the scope where it was created, even after that scope has finished executing.

This is not a trick or a quirk — it's a fundamental, intentional feature of JavaScript's design. Every function in JavaScript is a closure. Most of the time you don't notice it. When you do notice it, it becomes one of the most powerful tools you have.

Mental Model

Think of a closure as a function with a backpack. When a function is created, it packs up all the variables in its surrounding scope into that backpack. Wherever the function goes — passed as a callback, returned, stored in a variable — it carries that backpack with it.

simplest possible closure 
function makeGreeter(name) {
  // 'name' lives in makeGreeter's scope

  return function greet() {
    // greet() closes over 'name' — it remembers it
    console.log(`Hello, ${name}!`);
  };
}

const greetAli  = makeGreeter('Ali');
const greetSara = makeGreeter('Sara');

greetAli();   // "Hello, Ali!"
greetSara(); // "Hello, Sara!"

// makeGreeter has finished running. But greetAli still
// remembers 'Ali', and greetSara still remembers 'Sara'.
// Each closure has its OWN private copy of 'name'.
Key Insight The outer function (makeGreeter) has finished executing — it's gone. But the inner function (greet) keeps the variables alive in memory because it still references them. This is closure.

The Scope Chain — Where JS Looks For Variables

When a function needs to find a variable, JavaScript walks up the scope chain — looking first in the local scope, then outward, layer by layer, until it finds it or hits the global scope.

🌍 GLOBAL SCOPE
└── can see: everything at global level

function outerFunction() {
📦 OUTER SCOPE (has: outerVar)
└── can see: outerVar + globals

function innerFunction() {
🔍 INNER SCOPE (has: innerVar)
└── can see: innerVar + outerVar + globals
}
}

The chain only goes one direction — inward-out. Inner functions can see outer scopes. Outer functions cannot see inner scopes.

scope chain in action 
let globalVar = 'global';

function outer() {
  let outerVar = 'outer';

  function inner() {
    let innerVar = 'inner';

    console.log(innerVar);   // ✅ found in local scope
    console.log(outerVar);   // ✅ found in outer scope (closure)
    console.log(globalVar);  // ✅ found in global scope
  }

  // console.log(innerVar)  // ❌ ReferenceError — outer can't see inner
  inner();
}

outer();
Why the original example broke When you defined innerFunction outside of outerFunction and just passed it in, its scope chain was built at the global level. So it could never see outerVar, no matter where it got called. Scope is set at definition time, not call time.

How Closures Work Under The Hood

When JavaScript creates a function, it attaches a hidden [[Environment]] property to it — a reference to the scope (called a Lexical Environment) that existed when the function was defined.

When the outer function finishes, its execution context is popped off the call stack. Normally this would allow its local variables to be garbage collected. But if an inner function still holds a reference to that [[Environment]], those variables stay alive in memory.

each call creates a fresh environment 
function makeCounter() {
  let count = 0;  // private to this invocation

  return {
    increment: () => ++count,
    decrement: () => --count,
    value:     () => count,
  };
}

const counterA = makeCounter(); // new environment, count = 0
const counterB = makeCounter(); // new environment, count = 0

counterA.increment(); // 1
counterA.increment(); // 2
counterB.increment(); // 1  ← completely separate count

console.log(counterA.value()); // 2
console.log(counterB.value()); // 1
Critical Insight Every call to makeCounter() creates a brand new Lexical Environment with its own count. The closures returned from each call share their own environment — not each other's.

// Closures share the SAME environment

When multiple closures are created inside the same function call, they all share the same Lexical Environment. This is important:

shared environment between sibling closures 
function makeWallet(initial) {
  let balance = initial;

  const deposit  = (n) => { balance += n; };
  const withdraw = (n) => { balance -= n; };
  const check    = ()  => balance;

  // deposit, withdraw, and check ALL share the same 'balance'
  return { deposit, withdraw, check };
}

const wallet = makeWallet(100);
wallet.deposit(50);   // balance = 150
wallet.withdraw(30);  // balance = 120
wallet.check();        // 120

Real-World Closure Patterns

// 1. Function Factories pattern

Create specialized versions of a general function:

function factory
function multiplier(factor) {
  return (n) => n * factor;
}

const double = multiplier(2);
const triple = multiplier(3);

double(5);  // 10
triple(5);  // 15

// 2. Private State (Module Pattern) pattern

Simulate private variables — JavaScript has no private keyword in older code, but closures give you the same thing:

module pattern
const UserStore = (function() {
  let _users = [];  // private — no outside access

  return {
    add(user)    { _users.push(user); },
    remove(id)   { _users = _users.filter(u => u.id !== id); },
    getAll()     { return [..._users]; }, // return copy, not reference
    count()      { return _users.length; },
  };
})(); // IIFE — immediately invoked

UserStore.add({ id: 1, name: 'Ali' });
UserStore.count(); // 1
UserStore._users;  // undefined — truly private

// 3. Memoization pattern

Cache expensive function results using closure to hold the cache:

memoize
function memoize(fn) {
  const cache = new Map(); // closed over by the returned fn

  return function(...args) {
    const key = JSON.stringify(args);

    if (cache.has(key)) {
      console.log('cache hit');
      return cache.get(key);
    }

    const result = fn(...args);
    cache.set(key, result);
    return result;
  };
}

const expensiveCalc = memoize((n) => n * n * n);
expensiveCalc(10); // computed: 1000
expensiveCalc(10); // cache hit: 1000

// 4. Partial Application pattern

partial application
function partial(fn, ...presetArgs) {
  return function(...laterArgs) {
    return fn(...presetArgs, ...laterArgs);
  };
}

function add(a, b, c) { return a + b + c; }

const add10 = partial(add, 10);
add10(5, 3); // 18  (10 + 5 + 3)

// 5. Event Handlers & Callbacks pattern

You use closures every time you write a React onClick, an event listener, or a setTimeout:

closures in event handlers
function attachHandler(buttonId, message) {
  document
    .getElementById(buttonId)
    .addEventListener('click', () => {
      // This arrow fn closes over 'message'
      console.log(message);
    });
}

attachHandler('btn-1', 'First button clicked');
attachHandler('btn-2', 'Second button clicked');
// Each handler remembers its own message

The Gotchas — Where Everyone Gets Burned

// Gotcha #1: The Classic Loop Bug gotcha interview

This is the most common closure interview question. It trips up everyone the first time:

❌ broken with var
for (var i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);
  }, 1000);
}
// Prints: 3, 3, 3
// NOT 0, 1, 2 !
✅ fixed with let
for (let i = 0; i < 3; i++) {
  setTimeout(() => {
    console.log(i);
  }, 1000);
}
// Prints: 0, 1, 2 ✅
Why This Happens var is function-scoped, not block-scoped. All three callbacks close over the same i variable. By the time the timeouts fire, the loop has finished and i === 3. let creates a new binding per loop iteration, so each closure gets its own i.

The old-school fix before let existed was an IIFE:

✅ old-school fix using IIFE
for (var i = 0; i < 3; i++) {
  (function(j) {        // IIFE creates a new scope, capturing j
    setTimeout(() => {
      console.log(j);  // j is a local copy of i
    }, 1000);
  })(i);
}

// Gotcha #2: Closures Close Over Variables, Not Values gotcha

variable reference vs value
function example() {
  let x = 10;

  const getX = () => x; // closes over the VARIABLE x, not the value 10

  x = 99;               // mutate x after closure was created

  return getX;
}

example()();  // 99, not 10 !
// The closure sees the CURRENT value of x when it runs, not when it was defined
Rule Closures capture the variable binding, not a snapshot of the value. If the variable changes, the closure sees the new value when it runs.

// Gotcha #3: Definition Location vs Call Location gotcha

This is literally what broke your original code. Worth repeating explicitly:

where defined ≠ where called
let secret = 'outer';

function outer() {
  let secret = 'inner';
  logSecret(); // call from inside outer()
}

function logSecret() {
  // defined at global scope — sees global 'secret'
  console.log(secret);
}

outer(); // logs 'outer' — NOT 'inner' !
// logSecret was defined globally, so that's its scope

// Gotcha #4: this Is Not Closed Over gotcha

❌ this is lost in regular fn
function Timer() {
  this.seconds = 0;

  setInterval(function() {
    this.seconds++; // ❌ wrong 'this'
  }, 1000);
}
✅ arrow fn inherits this
function Timer() {
  this.seconds = 0;

  setInterval(() => {
    this.seconds++; // ✅ correct
  }, 1000);
}
Arrow Functions & this Arrow functions don't have their own this — they inherit it from where they were defined. Regular functions get their own this based on how they're called. This is why arrow functions are preferred for callbacks inside methods.

Memory, Performance & Leaks

Closures keep their outer scope alive in memory. This is a feature — but it becomes a memory leak when you hold references longer than needed.

❌ memory leak example
function attachHandler() {
  const hugeData = new Array(1000000).fill('x'); // 1MB of data

  document.getElementById('btn').addEventListener('click', () => {
    console.log('clicked');
    // This closure keeps hugeData alive FOREVER
    // even though we never use hugeData here!
  });
}

// Fix: only close over what you actually need
function attachHandlerFixed() {
  const hugeData = new Array(1000000).fill('x');
  const summary = hugeData.length; // extract only what you need
  hugeData = null; // allow GC to clean up

  document.getElementById('btn').addEventListener('click', () => {
    console.log(summary); // only closes over 'summary'
  });
}
Rule of Thumb Only close over what you actually use. If a closure captures a large object but only needs one property from it, extract that property first and close over the small value instead.

// When to be careful

Watch for closures in these scenarios: event listeners on long-lived DOM elements, setInterval callbacks that are never cleared, caches that grow without bounds, and closures stored in global variables.

Senior Dev Interview Q&A

// Q: "What is a closure?"

Senior Answer

"A closure is a function that retains access to its outer lexical scope even after the outer function has returned. In JavaScript, every function is technically a closure because they all have a [[Environment]] reference. The practical value is that closures let you create private state, factory functions, and callbacks that carry context — things like memoization, the module pattern, and event handlers all rely on closures."

// Q: "What's the difference between closure and scope?"

Senior Answer

"Scope is the set of rules that determine where a variable is accessible — it's a static concept defined by where code is written. Closure is the mechanism by which a function retains access to variables from its outer scope even after that scope's execution context is gone. Scope defines what's visible; closure is what makes that visibility persist."

// Q: "How would you implement a once() function?"

A function that only executes once no matter how many times it's called — a classic closure pattern used in real codebases:

once() — senior implementation
function once(fn) {
  let called = false;
  let result;

  return function(...args) {
    if (!called) {
      called = true;
      result = fn.apply(this, args); // preserve 'this' context
    }
    return result; // always return the same result
  };
}

const init = once(() => console.log('initialized!'));
init(); // "initialized!"
init(); // nothing
init(); // nothing

// Q: "Explain the loop + setTimeout bug and three ways to fix it"

3 fixes — know all three
// Fix 1: Use let (modern, preferred)
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}

// Fix 2: IIFE (pre-ES6 classic)
for (var i = 0; i < 3; i++) {
  (function(j) {
    setTimeout(() => console.log(j), 0);
  })(i);
}

// Fix 3: bind or factory
for (var i = 0; i < 3; i++) {
  setTimeout(console.log.bind(null, i), 0);
}

The Closure Cheatsheet

Definition

A function that retains access to its outer lexical scope after the outer function has returned.

Scope is lexical

Determined by WHERE code is written, not where it's called. Defined at creation time, not call time.

Captures variables, not values

Closures hold a reference to the variable binding. If the variable changes, the closure sees the new value.

Each call = new environment

Every invocation of the outer function creates a brand-new, independent Lexical Environment.

var vs let in loops

var is function-scoped → all closures share one variable. let is block-scoped → each iteration gets its own binding.

this is NOT closed over

Regular functions get their own this. Arrow functions inherit this from where they were defined.

Memory implications

Closed-over variables stay in memory as long as the closure exists. Only capture what you actually need.

Common patterns

Module pattern, function factories, memoization, partial application, once(), event handler callbacks.

You Now Think In Closures Every callback you write is a closure. Every React hook relies on closures. Every time you write a function inside a function, you're using closures. They're not a special trick — they're the everyday fabric of JavaScript.
